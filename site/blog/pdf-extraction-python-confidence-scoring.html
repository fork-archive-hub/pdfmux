“67% Confidence” — What Happens When Your PDF Extractor Actually Tells You It’s Failing

March 6, 2026 · 8 min read benchmark real-world test

Most PDF extraction tools demo well on clean, digital documents. A research paper. A nicely typeset invoice. The kind of PDF that every tool handles fine.

If you’re building a RAG pipeline and need reliable PDF extraction in Python, the real test isn’t whether a tool handles clean documents — it’s what happens when extraction partially fails.

We threw 4 real-world PDFs at pdfmux — a salary payslip with nested tables, a 12-page legal contract, a scanned bank form in two languages, and a 47-page startup pitch deck where half the text is baked into images. Here’s what we found.

The short version: pdfmux extracted all 4 PDFs with zero configuration. The pitch deck initially came back at 67% confidence with 24 empty pages. That confidence score — a feature no other open-source PDF tool offers — told us exactly which pages needed OCR. Switching to high quality recovered all 47 pages. 76 total pages processed in under 30 seconds.

Skip to the scanned bank form for the first failure case. Skip to the pitch deck for the real stress test.

The Test Setup

We installed pdfmux v1.0.1 from PyPI and configured it as an MCP server inside Claude Code — meaning an AI agent could call it directly as a tool, with zero human intervention.

pip install pdfmux

Three tools became available:

  • convert_pdf — extract text from any PDF, auto-detecting the best method
  • analyze_pdf — quick triage without full extraction (page count, type, per-page quality)
  • batch_convert — process an entire directory of PDFs

No configuration files. No API keys. No model selection. Just a file path.

The first two PDFs — a 5-page set of salary payslips with nested tables and a 12-page convertible note with deep clause numbering — extracted at 100% confidence in under 1 second each on standard quality.

Why this matters for RAG: If your extractor loses clause numbering or flattens defined terms, an AI agent referencing “Clause 4.2.2” will hallucinate. If tables become paragraphs, downstream parsing breaks. pdfmux preserved the full hierarchy and structure.

These aren’t the interesting cases. Every halfway-decent PDF extractor handles digital, text-layer PDFs. The question is what happens when they don’t.

The First Failure: A Scanned Bank Form in Two Languages

PDF type: Scanned/image-based, mixed language (English + Arabic) Quality: Standard → failed → High → recovered

A 12-page credit card application form from a major international bank. The document was scanned — meaning the text exists only as pixels, not as a text layer.

Standard quality result: 0 extractable text across all 12 pages.

This is actually the correct behavior. pdfmux detected the PDF was image-based and honestly reported:

Per-page quality: all 12 pages flagged as "empty"

Rather than returning garbage, it told us exactly what was wrong. This is pdfmux’s confidence scoring — the feature that no other open-source PDF tool offers. Most extractors return an empty string here and your downstream code has no idea anything went wrong.

Why pdfmux Works on Real-World PDFs

Self-Healing PDF Extraction Through Smart Routing

pdfmux doesn’t use one extraction method for everything. It runs a per-page classifier that checks text-layer density, embedded image count, and font metadata to categorize each page as digital, scanned, mixed, or graphical — then routes to the optimal extractor for that page type.

For the salary slips, it used the fast digital path (PyMuPDF). For the scanned bank form, it activated OCR. For the pitch deck, it used a hybrid approach: fast extraction for text-layer pages, OCR for image pages.

This is why it’s called pdfmux — it multiplexes between extraction strategies. And it’s why pdfmux doesn’t compete with PyMuPDF, Docling, or Marker — it uses them, routing each page to whichever tool handles it best.

PDF Confidence Scoring: The Feature No Other Open-Source Tool Has

Every extraction returns a confidence score. When the pitch deck came back at 67%, that wasn’t a failure — it was a signal. “I got what I could, but 24 pages need a different approach.”

The confidence-then-retry pattern is what makes self-healing extraction possible:

from pdfmux import process

result = process("pitch-deck.pdf")
print(f"Confidence: {result.confidence}")   # 0.67
print(f"Empty pages: {len(result.empty_pages)}")  # 24 of 47

# Confidence score tells you when to retry
if result.confidence < 0.8:
    result = process("pitch-deck.pdf", quality="high")
    print(f"Confidence: {result.confidence}")   # 0.95
    print(f"Empty pages: {len(result.empty_pages)}")  # 0 of 47

Eight lines. That’s the entire pdfmux story.

Quality Presets as a Dial

Instead of configuring OCR engines, model endpoints, and extraction parameters, pdfmux gives you three presets:

  • fast — digital text only, milliseconds per page
  • standard — smart routing with fallbacks (default)
  • high — OCR + all available extractors, maximum accuracy

One parameter. That’s it.

The Honest Limitations

  1. Arabic OCR — Tesseract’s Arabic model produced noisy output on the scanned bank form’s RTL pages. Usable for search indexing, not for direct display.
  2. Design-heavy slides at standard quality — If text is rendered as an image, standard quality can’t extract it. You need high. This is by design — OCR is slower and you shouldn’t pay that cost on every PDF.
  3. Handwritten text — Not tested here, but OCR-based extraction of handwriting remains an industry-wide challenge.

Arabic OCR and vision-based extraction are on the roadmap for v1.1. If you hit other edge cases, file an issue.
